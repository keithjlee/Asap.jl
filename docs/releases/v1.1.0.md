# Asap v1.1.0

Performance and solver-infrastructure release: a much faster mutating solve
path, pluggable linear-solver backends via LinearSolve.jl, per-axis
constraints for FDM form-finding, and two staleness/allocation bug fixes.

## Performance

- **Truss stiffness kernel fixed** — the kernel built its 6×6 block via
  `hvcat`, which silently fell back to heap-allocated matrices. Assembly of
  an 1,800-element spaceframe: 3.2 ms → **0.18 ms** (Julia 1.12) and
  10 MB → **48 bytes**; `solve!` 9.7 → 2.1 ms. This also erases the
  apparent Julia 1.11 → 1.12 regression (both versions now identical, and
  both faster than the old 1.11 numbers). A regression test pins
  allocation-free assembly for truss AND frame groups.
- **Refactorization reuse** — repeated `solve!` calls now rerun only the
  numeric factorization phase on the frozen sparsity pattern (symbolic
  analysis reused), on every backend. Mutate-and-resolve loops get this for
  free.
- `solve_cases!` performs one multi-RHS backsolve instead of per-case
  solves.

## Solver backends (new)

The built-in solver (CHOLMOD Cholesky, LDLᵀ fallback) remains the
zero-dependency default. Loading [LinearSolve.jl](https://github.com/SciML/LinearSolve.jl)
unlocks its full algorithm collection through one keyword:

```julia
using LinearSolve
solve!(model; solver = KLUFactorization())
solve!(model; solver = KrylovJL_CG())     # iterative — large models
solve!(model)                             # choice remembered on the model
```

Works in `solve_cases!` too. Notes: generic element types (`Model{BigFloat}`,
duals) can now use LinearSolve's generic direct solvers (the built-in CHOLMOD
path is Float64-only); iterative solvers make differentiable-path gradients
inexact adjoints (accuracy follows solve tolerance).

`solve`/`solve!` now extend the
[CommonSolve](https://github.com/SciML/CommonSolve.jl) verbs (a new,
featherweight dependency), so `using Asap, LinearSolve` — or any other
SciML package — shares one `solve!` binding instead of shadowing Asap's
API with an ambiguity error.

## FDM: per-axis constraints (new)

`FDMnode` fixity is now per-axis (`true` = free, matching `Node`):

```julia
FDMnode([x, y, z], Bool[false, false, true])   # plan prescribed, height free
```

FDM equilibrium is separable per coordinate, so mixed networks solve each
axis against its own free/fixed partition — prescribe the plan and let the
height find equilibrium (vault form-finding), or any other combination.
Uniform networks keep the single batched multi-RHS solve. Reactions report
per fixed axis. The FDM types also gained full docstrings and REPL displays.

## Bug fixes

- **Spring stiffness values were baked into the analysis cache at
  `process!` time** — replacing a `NodalSpring` between solves silently
  used the old value, contradicting the documented contract. Spring values
  are now read fresh at every assembly (both the in-place and
  differentiable paths).
- `to_network` no longer solves converted trusses with over-collapsed
  anchor semantics changing silently — supports become full anchors with
  the reasoning documented (partial anchors on converted trusses produce
  singular force-density systems; build networks with per-axis fixity
  directly when intended).

## Breaking

- `Network` field types tightened (`q::Vector{Float64}`, `P/Pn/xyz::Matrix{Float64}`)
  and `FDMnode.dof::Bool` replaced by `fixity::SVector{3,Bool}` — the
  `Bool` constructors still work (`FDMnode(pos, true)`); code reading
  `.dof` should read `all(node.fixity)`.
- `AnalysisCache.factorization` now holds a `FactorizationCache` wrapper
  (still solves via `\`); `cache.spring_nz` value pairs became
  `cache.spring_entries` index triples.
